ARCHITECTURAL PROBLEMS 🟠
Problem 1: No Alembic Migrations
alembic==1.13.1 is in requirements.txt but never initialized. You're using Base.metadata.create_all() in migrate.py which is fine for initial setup but doesn't support schema evolution. When you add a column (e.g., the market_cap column), you're doing it with raw ALTER TABLE SQL in reseed.py. This is how databases get out of sync between environments.
The reseed.py at project root is a red flag for any senior engineer reviewing this. It mixes schema migration logic with data seeding logic in an ad-hoc script. Delete it.
Problem 2: Manual Migration Script
migrate.py at the project root runs raw ALTER TABLE SQL. This:

Has no version tracking
Can't be rolled back
Can't be applied selectively
Breaks in production if run twice (some statements aren't idempotent)

You have Alembic. Use it.
Problem 3: Inconsistent Router Registration
In main.py:
pythonapp.include_router(funds.router, prefix="/api")      # funds.py: prefix="/funds" → /api/funds
app.include_router(portfolio.router, prefix="/api")   # portfolio.py: prefix="/portfolio" → /api/portfolio
app.include_router(tax.router)                        # tax.py: prefix="/api/tax" → /api/tax
The tax router embeds its own /api prefix while the others get it injected from main.py. This works but is inconsistent. Someone adding a new router will follow the wrong pattern.
Problem 4: Hardcoded CORS Origins
Production frontend URL is "https://portfoliox.vercel.app" which is a placeholder. When you actually deploy to Vercel the URL will be different and CORS will block everything.
Problem 5: reseed.py Exposes Data Model Assumptions
The root-level reseed.py uses raw SQL with hardcoded scheme codes that are assumed/guessed (the comments say "Assumed code"). If those codes are wrong in the database (which is likely since AMFI codes change), the holdings data is attached to the wrong funds. The find_codes.py script exists precisely because the seeder is using guessed codes — this is a data integrity problem.
