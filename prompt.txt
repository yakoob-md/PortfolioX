OBJECTIVE:
Set up the complete development foundation for PortfolioX — an Indian mutual fund 
portfolio analytics platform. This is Phase 0: Infrastructure & Data Foundation.

- Backend: Python FastAPI

STEP 4: DATABASE SCHEMA
Create these tables using SQLAlchemy declarative models (not raw SQL):

Table: funds
- scheme_code: String(10), primary key
- scheme_name: Text, not null
- amc_name: Text, not null  
- category: Text, nullable
- sub_category: Text, nullable
- plan_type: Text, nullable (values: 'Direct', 'Regular')
- option_type: Text, nullable (values: 'Growth', 'IDCW', 'Bonus')
- nav: Numeric(12,4), nullable
- nav_date: Date, nullable
- expense_ratio: Numeric(5,3), nullable
- aum_crore: Numeric(15,2), nullable
- launch_date: Date, nullable
- is_active: Boolean, default True
- created_at: DateTime with timezone, server default now()
- updated_at: DateTime with timezone, server default now()

Table: fund_holdings
- id: Integer, primary key, autoincrement
- scheme_code: String(10), foreign key to funds.scheme_code
- disclosure_date: Date, not null
- stock_isin: String(20), nullable
- stock_name: Text, not null
- sector: Text, nullable
- holding_percentage: Numeric(6,3), nullable
- market_value_cr: Numeric(15,2), nullable
- asset_type: Text, nullable (Equity/Debt/Cash/Other)
- created_at: DateTime with timezone, server default now()
- Unique constraint on (scheme_code, disclosure_date, stock_isin)

Create indexes on:
- fund_holdings.scheme_code
- fund_holdings.disclosure_date  
- fund_holdings.stock_isin
- funds.amc_name

STEP 5: data/scripts/load_amfi_schemes.py
Write an async Python script that:
1. Fetches the AMFI NAVAll.txt file from: https://www.amfiindia.com/spages/NAVAll.txt
2. Parses it — the format is:
   - Lines starting with a number followed by semicolons are fund data
   - Format: SchemeCode;ISINDivPayOut;ISINDivReinvestment;SchemeName;NAV;Date
   - Section headers (like "Open Ended Schemes(Equity)") indicate category
   - AMC names appear as standalone lines before the fund entries
3. For each fund entry, extract:
   - scheme_code, scheme_name, nav, nav_date
   - amc_name (from the section header above it)
   - plan_type: if "Direct" in scheme_name → 'Direct', else 'Regular'
   - option_type: if "Growth" in scheme_name → 'Growth', elif "IDCW" in scheme_name → 'IDCW', else 'Other'
4. Upsert all records into the funds table (insert or update on conflict)
5. Print summary: total funds loaded, AMCs found, any errors

The NAVAll.txt file has ~20,000 entries. Parse all of them.

STEP 6: db/repositories/fund_repo.py
Create a FundRepository class with these async methods:
- search_funds(query: str, limit: int = 10) → List[Fund]
  - Use PostgreSQL ILIKE for case-insensitive search on scheme_name
  - Also search on amc_name
  - Filter to is_active = true
  - Order by: exact matches first, then partial matches
- get_fund_by_code(scheme_code: str) → Fund | None
- get_funds_by_codes(scheme_codes: List[str]) → List[Fund]
- get_all_amcs() → List[str]

STEP 7: services/fund_resolver.py
Create a FundResolver service that:
- Takes a fund name string (user input, possibly fuzzy)
- Returns the best matching scheme_code and Fund object
- Uses the repository's search function
- Also tries to handle common abbreviations: 
  "PPFAS" → "Parag Parikh", "HDFC TF" → "HDFC Tax"

STEP 8: routers/funds.py
Create these API endpoints:

GET /api/funds/search?q={query}&limit={limit}

- Returns: List of {scheme_code, scheme_name, amc_name, nav, plan_type, category}
- Validates: query must be at least 2 characters
- Cache results in Redis for 1 hour

GET /api/funds/{scheme_code}
- Returns full fund details
- Cache in Redis for 24 hours

GET /api/funds/amcs
- Returns list of all unique AMC names
- Cache in Redis for 24 hours

CODING RULES:
- All database operations must be async (use async/await throughout)
- All external HTTP calls must use httpx AsyncClient, not requests
- Never use synchronous database operations
- Use Pydantic v2 models for all request/response shapes
- Use Python type hints everywhere
- No hardcoded values — everything configurable via env vars
- Handle HTTP errors gracefully (fund data APIs can be unreliable)
- Log errors but don't crash on non-critical failures (e.g., one fund fails to parse)
- Every function must have a docstring
